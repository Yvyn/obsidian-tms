## 1. Структура тест-плану

Тест-план — це звичайний `.md` файл в Obsidian. Плагін читає його структуру і перетворює на тест-ран.

### Заголовки-секції

Рядки що починаються з `#`, `##`, `###` тощо (без відступу) — це **секції**, не тест-кейси. Вони групують кейси і відображаються як заголовки в тест-рані.

```
## Авторизація
Вхід з валідними даними
Вхід з невірним паролем
Відновлення пароля

## Профіль користувача
Редагування імені
Завантаження аватара
```

> Заголовок включається в тест-ран лише якщо під ним є хоча б один обраний тест-кейс.

### Тест-кейси

Будь-який рядок що не є заголовком — тест-кейс. Вкладеність визначається відступами (tab або 2 пробіли).

```
## Кошик
Додати товар до кошика
  Перевірити лічильник
  Перевірити суму
Видалити товар
Оформити замовлення
```

> Заголовок з відступом (`	## щось`) — це вкладений тест-кейс, а не секція.

### Повний приклад файлу

```markdown
## Авторизація
Вхід з валідними даними @smoke @regression
Вхід з невірним паролем @regression
Вхід через Google @oauth

## API
GET /users @api @smoke
POST /users @api
DELETE /users @api @destructive
  Перевірити відповідь 204
  Перевірити що запис видалено з БД
```

## 2. Атрибути @

Атрибути — це теги у форматі `@назва` що додаються в кінець рядка тест-кейсу. Використовуються для фільтрації при генерації тест-рану.

```
Вхід з валідними даними @smoke @regression
GET /users @api @smoke
```

**Правила:**
- Починаються з `@`
- Можуть містити букви, цифри, `_`, `-`
- Регістр не важливий (`@Smoke` = `@smoke`)
- Кілька атрибутів на одному рядку — через пробіл

---

## 3. Автодоповнення атрибутів

При введенні `@` в будь-якому `.md` файлі — з'являється список підказок з усіма атрибутами знайденими у всіх файлах vault.

**Навігація:**
- `↑` / `↓` — переміщення по списку
- `Enter` — вставити обраний атрибут
- `Esc` — закрити підказку

## 4. Генерація тест-рану

Відкрийте тест-план і запустіть команду **Test Run** (через палітру команд, ribbon або status bar).

### Крок 1 — Вибір атрибутів

Відкривається модальне вікно з усіма атрибутами файлу.

| Дія | Стан | Колір |
|-----|------|-------|
| Один клік | Include | Зелений |
| Два кліки | Exclude | Червоний |
| Три кліки | Нейтральний | Сірий |

- **Include All** — позначити всі атрибути як include
- **Reset All** — скинути всі до нейтрального стану

**Логіка фільтрації:**
- Include: OR — кейс потрапляє якщо має хоча б один з включених атрибутів
- Exclude: кейс виключається якщо має хоча б один з виключених атрибутів
- Без фільтру (всі нейтральні) — обираються всі кейси

### Крок 2 — Review тестів

Список всіх тест-кейсів з секціями. Кейси що відповідають фільтру — попередньо відмічені.

- Вручну увімкнути або вимкнути будь-який кейс
- Кейси з вкладеними елементами мають кнопку `▶` для розгортання
- Лічильник показує `N / Total обрано`
- **← Back** — повернутись до вибору атрибутів
- **Generate Test Run** — створити файл тест-рану

### Крок 3 — Результат

Створюється новий `.md` файл у налаштованій папці з назвою:
```
{Назва файлу} - Test Run {дата-час}.md
```

Структура файлу:
```markdown
# Test Run: Назва проєкту
**Date:** 18.04.2026, 12:00:00
**Include:** @smoke @api
**Total Cases:** 5

---

## Авторизація

- [ ] **Вхід з валідними даними** @smoke @regression
- [ ] **GET /users** @api @smoke

---
```

## 5. Проведення тестування

В згенерованому тест-рані замінюйте `[ ]` на статус кейсу:

| Символ | Статус | Мітка |
|--------|--------|-------|
| `[x]` або `[p]` | Pass | ✅ Pass |
| `[f]` | Fail | ❌ Fail |
| `[s]` | Skipped | ⏭️ Skipped |
| `[b]` | Blocked | 🚫 Blocked |
| `[ ]` | Not Run | — |

Мітка (`✅ Pass |`, `❌ Fail |` тощо) **додається автоматично** після зміни статусу — не потрібно нічого вводити вручну.

**Приклад:**
```markdown
- [p] ✅ Pass | **Вхід з валідними даними** @smoke
- [f] ❌ Fail | **GET /users** @api
- [s] ⏭️ Skipped | **DELETE /users** @api
```

## 6. Підрахунок результатів

Відкрийте тест-ран і запустіть команду **Results**. В кінці файлу з'явиться таблиця:

```markdown
## Test Results Statistics

| Status | Count | % |
| --- | --- | --- |
| ✅ Pass | 8 | 61.5% |
| ❌ Fail | 3 | 23.1% |
| ⏭️ Skipped | 2 | 15.4% |
| **Total** | **13** | |
```

Команду можна запускати повторно — таблиця оновлюється, не дублюється.

## 7. Налаштування

Відкрийте **Settings → Test Manager System**.

### Папка для тест-ранів

Вкажіть папку куди зберігатимуться згенеровані тест-рани. Залиште порожнім — файли зберігатимуться в корені vault.

Можна вписати шлях вручну або натиснути **Browse…** і обрати папку.

### Кнопки

Можна вмикати/вимикати окремі кнопки:

| Налаштування | Опис |
|---|---|
| Ribbon: Test Run | Іконка 🧪 на лівій панелі |
| Ribbon: Results | Іконка 📊 на лівій панелі |
| Status bar: Test Run | Кнопка `🧪 Test Run` в нижній панелі |
| Status bar: Results | Кнопка `📊 Results` в нижній панелі |

### Гарячі клавіші

Гарячі клавіші налаштовуються через **Settings → Hotkeys**, пошук `Test Manager`.
